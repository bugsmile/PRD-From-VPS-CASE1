# Rooted PRD v0.1 품질 검토 및 개선 리포트

## 1. 기존 작성본 검토 결과 차트 (As-Is)

| 품질 점검 항목 | 상태 | 진단 및 이슈 사항 |
| --- | --- | --- |
| **1. Outcome-KPI 연계** | ⚠️ | KPI가 정량화(ex. 월 2회 이하)되어 있으나, 이를 **어떤 시스템과 경로를 통해 측정할지(Data Source)**가 불명확함. |
| **2. AC (Given-When-Then)** | ⚠️ | 정상 해피 패스(Happy Path) 위주로 작성됨. 네트워크 단절, 반려동물 오인식, 백엔드 장애 등 **실패/예외 케이스가 일절 누락**됨. |
| **3. 비기능 요구사항 (NFR)** | ❌ | 성능 지연, 가용성 보장율, 에러율 등 핵심 임계치가 `[TBD]`로 비어있음. 시스템 모니터링 및 알림(Alert) 기준이 부재함. |
| **4. Differential Value** | ⚠️ | 대안 제품과의 오탐률 비교는 명확하나, 동선/인식의 물리적 정확도(Accuracy) 측정이 명시되지 않아 실질적 성능 비교가 어려움. |
| **5. Proof** | ⚠️ | 실험 가설은 정리되어 있으나, 측정할 테스트 집단의 규모(샘플 사이즈)와 검증 소요 기간, 구체적인 측정 시스템 도구가 명시되지 않음. |

## 2. 챕터/파트별 수정 방향

1. **[수정 방향] §1.3 성공 지표:**
   - 지표별 구체적 측정 경로(DB `is_false_alarm` 쿼리, Amplitude/Mixpanel 인앱 이벤트 트래킹, CS 티켓 태깅 시스템 등)를 명시하여 측정 주체를 명확히 함.
2. **[수정 방향] §3. 사용자 스토리와 수용 기준 (AC):**
   - 센서 오프라인 장애 상황(Heartbeat 유실), 반려동물로 인한 오탐 회피 조건, 외부 시스템(EMR) 연동 장애 시 재시도 등에 관한 '실패 케이스(Failure Cases)' AC들을 각각 추가하여 검증 커버리지 확대.
3. **[수정 방향] §5. 비기능 요구사항 (NFR):**
   - `[TBD]`로 남겨진 성능 지연(p95 응답속도) 및 신뢰성(SLA 99.9%, 오류율 등) 임계치를 업계 표준 스펙 수치로 정의.
   - PagerDuty/Datadog 등을 활용한 서비스 다운타임/오류 긴급 알림(Alert) 임계치 신규 추가.
4. **[수정 방향] §8. 실험·롤아웃 측량 (Proof):**
   - 실험 벤치마크 단계에서 실험군 규모(B2B 요양원 병상 수, B2C 가구 수), 기간(4주 등), 구체적 측정 프레임워크(Amplitude 리텐션 측정)를 세부 명시하여 실질적인 파일럿 테스트가 되도록 보완.

## 4. 리라이트된 PRD 부분 (수정된 섹션)

> **안내:** 기존 PRD 원문 중 품질 개선을 위해 수정이 필요한 항목(1.3, 3, 5, 8.2 단락 일부)만 리라이트 하였습니다.

### 1.3 성공 지표

- **북극성 지표:** 월간 오탐 빈도 ≤ 2회/가구 
  - **측정 경로:** 센서 DB의 `is_false_alarm` 플래그 누적 카운트 및 보호자 앱 내 '오탐 신고' 버튼 피드백 로그 월간 배치 분석
- **보조 KPI 1:** 앱 데일리 리포트 주간 확인 빈도 ≥ 5회/주 
  - **측정 경로:** 제품 애널리틱스(Amplitude/Mixpanel 등) 내 `view_daily_report` 이벤트 발생 유저 비율(WAU) 트래킹
- **보조 KPI 2:** 설치/운영 후 어르신 사용자 불만 및 기기 직접 조작 요인에 의한 이탈율 = 0건 
  - **측정 경로:** 고객센터/CRM 환불 및 해지 CS 티켓 사유 내 '설치 거부감/불편함' 태킹 누적 집계
